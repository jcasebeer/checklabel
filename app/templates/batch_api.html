Batch API

Check a few hundred labels in one request.

← Single check

Try it now

Download sample-batch.zip — four real label images from the TTB Public COLA Registry and a ready-to-submit manifest.json (one label is a front/back pair). 

unzip sample-batch.zip -d sample && cd sample

curl -X POST {{ base_url }}batch \
  -F 'manifest=<manifest.json' \
  -F 'files=@barolo-chinato.jpg' \
  -F 'files=@alvides.jpg' \
  -F 'files=@tx-panel1.jpg' \
  -F 'files=@tx-panel2.jpg'

The tx label is deliberately interesting: its registered brand (TX) appears inside a longer printed name, so it comes back needs review rather than a silent pass — that's the screening design working.

Request

POST /batch as multipart/form-data:

fieldvalue
filesOne part per label image (repeat the field).
manifestJSON object with the expected values for every uploaded file (shapes below).
modesync (default) or queued.

Manifest shapes

Key by filename for a single-image label, or by any label id with a files list for a multi-panel label — all panels of one label are analyzed together in a single pass:

{
  "stones_throw_red.jpg": { "brand": "Stone's Throw", "abv": 13.5 },
  "harbor-light":         { "brand": "Harbor Light", "abv": 6.2,
                            "files": ["hl_front.jpg", "hl_back.jpg"] }
}

abv is optional — if the application doesn't list one, that check is reported as skipped and doesn't block approval. Files without a manifest entry, and manifest entries without an uploaded file, are reported as inline errors — never silently dropped.

Two modes

sync (default)queued
HowLive model calls, bounded concurrencyOne Anthropic Message Batch
LatencyMinutes for a few hundred labelsUsually under an hour
Token costStandard50% of standard
ResponseResults in the response body202 + a batch_id to poll

Queued fits the peak-season drop: submit 300 labels at 5 pm, collect results in the morning at half the cost. Add -F 'mode=queued', then poll:

curl {{ base_url }}batch/msgbatch_XXXX

While processing you get a status payload; once ended, the same result shape as sync mode. If the server restarted while the batch was in flight, the poll returns 409 — resubmit the batch (nothing is persisted server-side by design).

Response

{
  "summary": { "total": 3, "passed": 2, "failed": 0, "needs_review": 1, "errored": 0 },
  "results": [
    {
      "file": "tx",
      "result": {
        "overall": "warn",          // pass | fail | warn | error
        "checks": [
          { "field": "Brand name", "status": "warn",
            "expected": "TX", "found": "TX Experimental Series ...",
            "detail": "The application's brand appears within the label text — please confirm by eye." },
          { "field": "Alcohol content", "status": "skipped", ... },
          { "field": "Government warning", "status": "pass", ... }
        ],
        "extracted": { ... }        // everything the model transcribed
      }
    }
  ]
}

Overall verdicts: any failing check fails the label; anything unconfirmed (near-miss brand, unverifiable bold header) is needs review for a human; skipped checks never block approval.
